## 项目概览

- 后端：Python、FastAPI、Uvicorn、SQLite，主要代码集中在 `server.py`。
- 前端：无构建步骤的原生 HTML、CSS 和 JavaScript，位于 `public/`。
- 持久化数据：`data/chat.sqlite3`；用户上传及同步的头像位于 `data/avatars/`。
- 部署：支持 Windows 批处理脚本和 Docker Compose，默认端口为 `8787`。

## 明确产品要求

### 产品定位与访问流程

- 项目名称为“暗号频道”，是一个可部署到云服务器的赛博风、异世界主题聊天室。
- 访客不能直接进入频道。必须先填写代号并选择头像，提交申请，等待管理员审核通过后才能聊天。
- 用户状态包括 `pending`、`approved`、`rejected`、`banned`；前端必须清楚反馈等待审核、已进入、被拒绝和被封禁状态。
- 身份以浏览器本地保存的设备码为基础。不得无故更换 `cyber-chat-device` 等现有本地存储键，否则老用户会丢失身份。
- 昵称最长 18 个字符，单条消息最长 420 个字符。
- 已批准用户可以进入公共频道，也可以从在线成员、消息头像或私聊列表发起一对一私聊。
- 公聊和私聊都要实时更新；当前实现以 SSE 为主，并保留轮询回退以兼容无法维持事件流的公网隧道。

### 管理员入口与权限

- 管理控制台内置在聊天室页面，不是独立后台站点。
- 管理入口必须对普通访客隐藏。只有地址带 `?admin=1` 时才加载/展示入口；旧 `/admin.html` 只负责跳转到 `/?admin=1`。
- 管理员登录后可在同一页面：审核或拒绝申请、查看用户、保护用户、封禁用户、查看并撤回公聊/私聊消息、清屏，以及管理头像库。
- 管理员登录成功时，当前设备可成为保护号；保护号不能被封禁。保护状态可由管理员开启或取消。
- 封禁以设备码为基础，并默认同时记录 IP 指纹；不要保存明文 IP。
- 管理员登录有失败次数限制，管理令牌有有效期。不得绕过现有认证依赖或把管理员口令暴露给前端。

### 消息与数据保留

- 聊天消息需要长期保存在 SQLite 中，不能因为刷新页面或服务重启而消失。
- 管理员撤回消息属于软撤回：数据库保留原记录，用户端只显示“已撤回”。公聊和私聊均遵守此规则。
- 管理员清屏只改变当前频道的可见起点，不物理删除历史消息。
- 历史记录当前最多返回 300 条；修改此限制前需确认性能和产品影响。
- `data/chat.sqlite3`、`data/avatars/` 和 `data/backups/` 是用户数据，不得在开发、测试或整理文件时覆盖、清空或提交其内容。

### 头像系统

- 头像选择页为 `/avatars.html`；选中结果保存在本机。如果本机已有代号，保存头像时还要同步服务器身份。
- 随项目发布的头像来自 `public/assets/avatars/`，服务启动时同步到数据目录，并与管理员上传的头像共同组成头像库。
- 管理员可以上传、改名、排序、启用或停用头像；系统必须始终至少保留一个启用头像。
- 上传只接受内容真实匹配的 PNG、JPG/JPEG 或 WebP，单文件不超过 2 MB，不能只信任扩展名或请求声明的 MIME 类型。
- 已有用户和历史消息的头像快照/引用要保持可显示，停用头像不能导致历史内容损坏。

### 界面与使用方式

- 保持现有赛博、黑红黄、高对比度的视觉语言和异世界氛围；除非用户明确要求，不要改成通用后台模板风格。
- 页面必须支持桌面和手机窄屏，关键操作不能只依赖悬停。
- 用户输入插入页面前必须转义，动态图片地址和标签也要按现有安全方式处理。
- 网站必须通过服务地址打开，例如 `http://127.0.0.1:8787/`；不要引导用户直接双击 `public/index.html`。
- 本地 Windows 使用批处理脚本启动；需要分享给同学时，可通过 Cloudflare Tunnel，缺失时使用 SSH/Serveo 回退。

### 部署与安全

- 本地预览允许 `ALLOW_DEV_DEFAULTS=1`；公网或生产环境禁止依赖默认管理员口令和默认密钥。
- 生产运行必须设置 `ADMIN_PASSWORD` 和足够长、随机的 `APP_SECRET`。
- 保持现有 CSP、禁止 iframe、MIME 嗅探保护、引用来源策略和权限策略等安全响应头。
- Docker 对外端口可由 `PORT` 调整，容器内部仍监听 `8787`；持久化数据使用 `/data` 数据卷。
- 公网部署建议在应用前使用提供 HTTPS 的反向代理或云平台入口。

## 常用命令

安装依赖：

```powershell
python -m pip install -r requirements.txt
```

本地启动：

```powershell
$env:ALLOW_DEV_DEFAULTS="1"
python server.py
```

生产方式启动前必须显式设置 `ADMIN_PASSWORD` 和 `APP_SECRET`，不要把真实密钥写入仓库。

基础语法检查：

```powershell
python -m py_compile server.py
```

Docker 配置检查：

```powershell
docker compose config
```

## 修改原则

- 修改前先阅读相关文件及其调用方，保持现有 API 字段名和前端存储键兼容。
- 后端改动尽量保持在 `server.py` 的现有 FastAPI/SQLite 架构内；不要无故引入框架或构建系统。
- 前端继续使用原生 JavaScript 和现有样式体系，避免新增包管理器或运行时依赖。
- 数据库结构变更必须兼容已有数据库；使用可重复执行的迁移逻辑，不得假定数据库为空。
- 保持安全响应头、管理员认证、登录限流、消息长度限制和头像类型/大小校验有效。
- 所有用户输入在进入 HTML 时必须安全转义；不要使用未经处理的输入拼接 SQL 或 DOM HTML。
- 不要修改或提交运行日志、缓存、视觉检查产物和本地数据库，除非任务明确要求处理这些文件。
- 不要覆盖 `data/chat.sqlite3`，也不要删除 `data/avatars/` 或 `data/backups/` 中的用户数据。
- 保持文本文件为 UTF-8。遇到中文显示异常时，先确认读取工具的编码，不要直接批量重写内容。

## 文件职责

- `server.py`：API、认证、SQLite 数据访问、事件推送和静态文件服务。
- `public/index.html`、`public/app.js`、`public/styles.css`：聊天主界面。
- `public/admin.js`、`public/admin-panel.js`：管理员功能。
- `public/avatars.html`、`public/avatars.js`：头像选择与管理界面。
- `public/assets/`：随应用发布的静态图片和头像素材。
- `Dockerfile`、`docker-compose.yml`：容器部署配置。
- `*.bat`：Windows 本地启动和公网穿透辅助脚本。

## 验证要求

- 每次 Python 改动后至少运行 `python -m py_compile server.py`。
- API 或数据库改动应使用临时数据库验证，禁止拿仓库中的正式 `data/chat.sqlite3` 做破坏性测试。
- 前端改动至少检查桌面和窄屏布局，并确认浏览器控制台无新增错误。
- 涉及身份流程时，覆盖访客申请、管理员审核、进入聊天室和封禁/拒绝等相关路径。
- 涉及消息功能时，覆盖普通消息、私聊、历史记录、撤回/清屏及事件流更新。
- 涉及头像时，验证默认头像、上传限制、启停、排序以及已选头像的兼容性。

## 提交前检查

- 只包含与当前任务相关的改动，不整理或覆盖用户已有的无关修改。
- 确认没有加入密码、令牌、隧道地址、日志内容、数据库或其他本地敏感数据。
